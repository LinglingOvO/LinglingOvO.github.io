Cursor使用

Qi2025-02-07
  1. Cursor Tab 代码自动补全与修改
  2. Cursor 中具有光标预测的功能

基于 Chat 和 Composer 的两种问答模式：

1. Chat 模式类似于 GPT 的 AI 聊天
2. COMPOSER模式会根据聊天的结果快速生成与修改现有代码
    1. 具有normal 和 agent 两种模式：
        1. nromal：可以检索代码库和文档，速度相对 agent 较快
        2. agent：除了创建文件，还可以自动提取相关上下文，运行终端命令按照你的语义去搜索代码、执行文件操作，但只支持claude模型
3. 快捷键：Ctrl + L、Ctrl + I
  1. Cursor 是基于项目的，打开哪个文件夹，那个文件夹就会被认为是一个项目，Cursor 会自动采集项目内的代码。
    1. 在聊天框中输入 @ 后会唤醒一个功能菜单，菜单中的功能本质上是为了辅助我们与 AI 更好地交流
      1. 比如想引用某个文件时，可以 @Files 然后选中某个文件
      2. 想引用某个文件夹时，可以 @Folders 然后选中某个文件夹，更方便的可以直接拖拽文件 / 文件夹到聊天框内
      3. @Code，可以直接选中一块代码
      4. 针对某个项目，想让 AI 根据某个文档生成一些东西，可以 @Docs 然后选中某个文档，除了 Cursor 本身附带的文档，我们可以添加自定义的文档链接并为其命名，选中这个文档之后 Cursor 会将文档内容纳入上下文
        1. 可以在设置中，Features 中的 Docs 自行添加，输入文档链接即可，但在添加链接时，如果是单纯的 URL，如https://docs.cursor.com，Cursor只会索引这一个页面的内容，如果在链接后加一个 /，如https://docs.cursor.com/，那么 Cursor除了索引这个链接文档的内容之外，还会索引这个文档之下所有的子页面和子目录
      5. 使用临时链接的方法：@链接，Add link 后
      6. 想要搜索网络上的内容时，可以输入 @Web，之后 Cursor 会根据你输入的信息和上下文构建搜索查询，搜索网络，查找相关的信息作为附加上下文，再给予回复，相当于一个 AI 搜索引擎的能力。
        1. 使用场景：比如用了一个新的技术，遇到了问题，可以把这个问题描述清晰后，跟一个 @Web，然后 Cursor 会自动查询网络上与你描述相符的信息，并给予答复。
      7. 也可以通过@git，来选择某个历史的 git 提交，将某一次提交添加到输入框中作为上下文，Cursor 会自动扫描我们对应提交中的代码，查看修改了哪些东西，也可以选中多个提交让 AI 去对比这几次提交的一些差异，来查找一些问题
      8. @Notepad：基本功能有 3 点，当临时笔记记录； 充当项目相关的一些上下文记录；保存一些 AI 的对话历史记录。在 Cursor 中 Chat 和 COMPOSER 两种模式的上下文是不互通的，而 Notepad 可以理解为 Cursor 针对这种情况做出的解，可以创建一个 Notepad，使用其记录开发思路或保存重要的代码片段或 AI对话记录，提供 Chat 和 COMPOSER 的沟通桥梁
      9. @Codebase：会采集代码中我们代码中重要的文件或者代码块， 在输入框输入 @Codebase 后，Crusor AI 会根据你给出的指令对采集的上下文进行排序然后推理，最后给出最匹配的答复。主要做了以下 4 件事情：
        1. 收集：Cursor 会扫描我们整个项目，查到与指令相关的文件或者是代码块
        2. 重新排序：Cursor 会根据与查询的相关性对上下文进行一个排序，简单理解就是收集到的这些文件和指令越相关，排的就会越靠前
        3. 推理：Cursor 会思考如何利用上下文
        4. 生成：给出回应

Cursor 会尊重 gitingore，Cursor 在索引前会检查你的项目根目录，如果你的根目录中有.gitignore 文件，那 .gitignore 文件中声明的文件或者是目录都不会被索引，但如果没有.gitignore 文件或者想要尽可能缩小一些索引的范围（减少索引的文件数量，让索引的数据尽可能小，以帮助代码生成理解会更准确），你可以在项目根目录创建一个.cursorignore 文件，来声明一些索引要忽略的文件或者是目录

那么哪些文件不应该被索引呢？比如项目中的日志文件（主要记录程序运行中的信息）、环境变量文件（配置项目的一些运行环境），这些文件存储着我们不想被其他人知道的一些数据信息。还有构建输出的一些文件，是项目构建过程中产生的一些文件，还有缓存文件（这种是针对项目本地运行优化而产生的一些缓存信息文件），第三方的一些配置文件（因为它单纯只是某些三方工具的配置，和我们项目的核心代码没有任何的关联），这些文件对 AI 的理解和生成没有什么帮助，所以没有必要纳入索引，可以直接忽略。

当你不知道怎么写 .innore文件时，可以在 COMPOSER 聊天框中让 AI 去帮你检索整个项目的目录，帮你生成一个 .cursorignore 文件就可以了

Cursor Rules：在设置中的 Rules for AI，这是 Cursor 的全局规则配置。除了全局规则，也可以在项目中创建一个 Cursor 规则，比如在项目的根目录中创建一个 .cursorrules 文件，然后在该文件中创建只针对这个项目的一些 Cursor 规则

在 COMPOSER 生成代码之后，可以通过 Accept all 进行全部应用，或 Reject all 进行全部取消。如果是多个文件的修改，在上方会有 Save all 按钮，其会先保存所有修改，然后可以通过运行这个项目去查看效果，如果符合需求就可以点击 Accept all 进行应用，也可以取消

如何回滚：在多次问答修改大量代码之后，想把代码恢复之前的版本，

Cursor 会给每次 Composer 的问答做一个 Checkpoint 标记，所以只需要在聊天框中找到某次提问，聊天上方会有一个 Checkpoint 标记，点击其右方的 restore 按钮，就可以进行恢复

如何提问？：提出的指令描述中某些词汇的表达让 AI 产生了误解，从而做出了错误的答复，就需要全面且清晰的表达，故可进行细化，从而进行规避，

  1. 让 AI 复述需求指令：在给出 AI 指令或者说问题时，我们要先让 AI 复述一遍我们的需求，再进行答复。以此判断 AI 是否准确理解了我们的需求。如：请你先复述一遍我的需求再进行答复，让我能够确认你真的理解了我的需求指令
  2. 明确需求辐射范围：在发出一个需求指令之前，要尽可能的让这个需求指令足够小，足够单一化，不要让 AI 猜你要做什么，要有足够的针对性，并且明确这次指令的范围。在 AI 编程中，在提出一个问题或者说需求指令时，应该尽可能的携带上相关的代码或者文件，并且告诉 AI 要在什么范围内做修改
  3. 需求拆解，
  4. 把 AI 当成小孩子

Cursor 整体使用思路：对需求不是很明确时，不应该让 COMPOSER 直接帮你写代码，而是先在 Chat 模式中和 AI 交流自己的想法，先描述你的需求，你想做什么事情，让 AI 对你的需求有个基本了解。之后可以把需求进行拆解，把这些通过 Notepad 单独记录下来，然后按照需求步骤在 COMPOSER 输入框中选择对应的 Notepad 去完成需求。在这个过程中，如果 COMposer 的生成有问题，可以再创建一个 Notepad 去记录问题，在 Chat 中选中这个 Notepad 去询问，然后更新，最后回到 COMPOSER 中去生成

已经有一个已知项目，想利用 Cursor 去辅助开发这个项目的后续需求，如改 BUG等，那么在用 Cursor 打开项目之后，

  1. 让 Cursor AI 检索整个项目，生成一个 .cursorignore 的文件，添加上你想要索引忽略掉的文件，之后在设置 -> Feartrues -> Codebase indexing 中重新索引
  2. 把项目相关的文档，如需求文档、接口文档、技术文档录入到 Docs 里以作备用
  3. 在项目根目录中，新建一个 .cursorrules 文件，去设置你的项目规则，当然如果不知道怎么写，可以在 Chat 模式中，@对应的文档然后让 AI 检索整个项目生成 rulse 文件

Chat 模式和 Composer 模式是相互独立的。

锻炼与 AI 的沟通能力，

白嫖 Cursor：

  1. 由于 Cursor 是邮箱注册，注册就会有 14 天的体验期，所以得出的策略是每 14 天用一个新的邮箱注册，但 Cursor 会对 API 是有限额的，
    1. 使用临时邮箱，进入一个临时邮箱平台后，无需注册，平台会自动分配给你一个邮箱账号，可以在一段时间内使用这个邮箱去收一些信息，但大多不能发信息，网站：tempmail.cn
    2. 无限邮箱： 如2925无限邮（2925.com），需要在无限邮箱平台注册一个邮箱账号，

Cursor 更新了用户指纹检测机制，即同一台设备使用免费账号太多次，会限制使用，
